OVERVIEW
--------
Central Python class (BaseStation) designed to run on a Raspberry Pi.
Managed node connections, WiFi/AP networking, Django server lifecycle,
and a separate Node.js webapp process. In practice, the Pi kept
crashing so everything was moved to a teammate's laptop, making most
of this code non-functional in deployment.

PROBLEMS
--------
- Hardcoded file paths (Diego's desktop, Pi-specific /etc/ paths)
- DIY JSON state persistence instead of a real database
- Bluetooth node connection never worked on the Pi (bluepy install failed)
- AP mode config writes directly to /etc/hostapd and /etc/dnsmasq
  (host OS level, not portable)
- Webapp launched via subprocess.Popen pointing to a hardcoded Node.js path
- systemd service file written at runtime (brittle, non-portable)
- install_dependencies() is redundant when using Docker
- No separation of concerns — one class doing networking, BT, server
  lifecycle, persistence, and webapp management

VERDICT
-------
REBUILD — Score 2/10. Almost entirely obsolete given Docker architecture.
Extract node ID validation logic only.

WHAT CARRIES FORWARD
--------------------
- authenticate_node() regex pattern (PDN#\d{6}) -> Django model validator
- connected_nodes concept -> PostgreSQL Node model
- State flags (server_running, access_point_mode, etc.) -> reference only
  for defining system state requirements
